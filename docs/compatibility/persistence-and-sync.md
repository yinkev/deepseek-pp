# Persistence and Sync Compatibility Contracts

Baseline: v1.10.0, commit `165ec46`. The production code owns 35 fixed `chrome.storage.local` keys, two fixed `chrome.storage.session` keys, and three IndexedDB databases. `core/platform/browser.ts` exposes an arbitrary-key adapter but has no production consumer; it is not approval for new unregistered keys.

For unversioned single-key records, the minimum future rule is: keep the key, preserve readable historical values, reject corrupt or explicitly future-versioned data before writing, and never overwrite original data with defaults merely because decoding failed. A table row calling out current defaulting/filtering describes present behavior, not permission to repeat silent loss.

## Sensitive-Data Note

`LS-003`, `LS-027`, `LS-028`, `LS-029`, and `LS-032` currently persist MCP secrets, API keys, authorization/client headers, or sync credentials in extension local storage. This registry records the compatibility surface; it does not claim at-rest secrecy. Any security migration must preserve recoverable user configuration, avoid logging secret values, and make destructive credential reset explicit.

## Current Evidence Limit

T1.4 adds raw historical fixtures for Memory v1-v3, Artifact legacy storage, Project v1/v2, Saved Items legacy/v1/future, Scenario storage, and all six sync JSON files. Fake IndexedDB tests execute the production Memory v1→v3 and v2→v3 upgrades, v3 project-scope reopen, and Artifact legacy migration. T2.4 adds an independent raw schema-v1 generation fixture, exhaustive upload boundaries, strict generation reads, legacy no-dual-write compatibility, and newest-live Google Drive canonical-object selection for duplicate names. T2.5 adds the raw schema-v1 local journal fixture, exact Memory/key preimage recovery, deterministic ID reuse, every-write fault injection, restart recovery, and idempotent retry evidence. Project v1 reset, Artifact filtered-row loss/no-retry/dual truth, Saved Items future-version downgrade, and Scenario broad fallback remain executable or enumerated migration gaps rather than legal compatibility outcomes.

Future database-version open behavior remains intentionally unimplemented here. T3.3 owns versioned domain codecs, migration, raw-data preservation outside the short-lived sync undo log, and authoritative persistence repositories. Generation garbage collection, distributed conflict resolution, and general concurrent user-write serialization remain outside T2.4/T2.5; orphaned remote objects and prepared local journals are never reported as committed success.
